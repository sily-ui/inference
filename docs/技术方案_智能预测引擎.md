### 3.5.1.2 ReAct 认知循环详解

ReAct 架构通过 **思考-行动-观察** 的循环机制，使 AI 能够像人类分析师一样逐步推理：

**1. Thought（思考）**
- 分析当前已掌握的信息
- 识别信息缺口
- 决定下一步需要获取什么数据
- 评估当前预测的可信度

**2. Action（行动）**
- 根据思考结果选择合适的工具
- 构造工具调用参数
- 执行工具调用

**3. Observation（观察）**
- 接收工具返回的结果
- 解析和结构化数据
- 将新信息整合到已有认知中

**4. 循环终止条件**
- 达到最小工具调用次数（2次）
- 达到最大工具调用次数（5次）
- 信息充分，可以输出最终结论

```python
# ReAct 循环核心逻辑示例
class PredictionAgent:
    MAX_TOOL_CALLS_PER_MODULE = 5
    MIN_TOOL_CALLS_PER_MODULE = 2
    
    def _run_react_loop(self, module_name: str, context: Dict) -> Dict:
        """执行 ReAct 认知循环"""
        tool_calls_count = 0
        used_tools = set()
        conversation_history = []
        
        for iteration in range(self.MAX_TOOL_CALLS_PER_MODULE):
            # 1. 调用 LLM 进行思考
            response = self.llm.chat(
                messages=conversation_history,
                system_prompt=ANALYSIS_SYSTEM_PROMPT
            )
            
            # 2. 解析思考和行动
            thought, action = self._parse_react_response(response)
            
            # 3. 如果是工具调用
            if action and action.type == "tool_call":
                result = self._execute_tool(action.tool_name, action.parameters)
                tool_calls_count += 1
                used_tools.add(action.tool_name)
                
                # 4. 构造观察反馈
                observation = self._format_observation(
                    action.tool_name, result, 
                    tool_calls_count, used_tools
                )
                conversation_history.append(observation)
                
            # 5. 如果是最终答案且满足最小调用次数
            elif action and action.type == "final_answer":
                if tool_calls_count >= self.MIN_TOOL_CALLS_PER_MODULE:
                    return self._parse_final_answer(action.content)
                else:
                    # 强制继续调用工具
                    conversation_history.append(
                        self._format_insufficient_tools_msg(tool_calls_count)
                    )
        
        # 达到最大次数，强制输出结论
        return self._force_final_answer(conversation_history)
```

### 3.5.1.3 OASIS 框架集成

OASIS 框架为预测引擎提供了 **多智能体社交模拟** 的底层能力：

**核心组件集成：**

| OASIS 组件 | 功能描述 | 在预测引擎中的作用 |
|-----------|---------|------------------|
| `Agent Profile Generator` | 基于知识图谱生成智能体人设 | 为模拟提供真实的社交参与者 |
| `Social Behavior Engine` | 模拟点赞、转发、评论等社交行为 | 生成可用于预测的时序数据 |
| `Memory System` | 管理智能体的记忆和认知状态 | 通过 Zep 实现时序记忆更新 |
| `Platform Simulator` | Twitter/Reddit 平台行为模拟 | 提供多平台舆情扩散数据 |

**OASIS 与 ReAct 的协同：**

```
┌────────────────────────────────────────────────────────────┐
│                    OASIS + ReAct 协同流程                   │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Step 1: 事件输入                                          │
│      ↓                                                     │
│  Step 2: OASIS 生成 Agent 群体（基于知识图谱实体）           │
│      ↓                                                     │
│  Step 3: OASIS 运行社交模拟（多轮次行为生成）                │
│      ↓                                                     │
│  Step 4: 模拟数据 → ReAct 预测引擎                          │
│      ↓                                                     │
│  Step 5: ReAct 循环分析（调用 SIR/贝叶斯等工具）            │
│      ↓                                                     │
│  Step 6: 生成预测报告（时间轴 + 情景 + 干预建议）            │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 3.5.1.4 多阶段状态管理

预测引擎采用状态机模式管理整个预测流程：

```python
class PredictionStage(Enum):
    """预测阶段状态机"""
    PENDING = "pending"       # 等待开始
    PLANNING = "planning"     # 策略规划
    ANALYZING = "analyzing"   # ReAct 分析
    PREDICTING = "predicting" # 生成预测
    REFLECTING = "reflecting" # 结果反思
    COMPLETED = "completed"   # 完成
```

每个阶段的详细日志通过 `PredictionLogger` 记录到 `agent_log.jsonl`，支持完整的可追溯性。

### 3.5.3.2 SIR 传播模型详解

SIR（Susceptible-Infected-Recovered）模型是预测引擎用于舆情传播趋势分析的核心算法工具。该模型借鉴传染病动力学原理，将人群划分为三类状态：

**模型原理**

SIR模型通过微分方程描述三类人群的动态变化：

- **易感人群（S）**：尚未关注事件但可能关注的人群，其变化率受传播率和恢复人群回流影响
- **感染人群（I）**：当前关注事件的人群，直接对应舆情"热度"指标，其变化率由新关注人数和遗忘人数共同决定
- **恢复人群（R）**：曾经关注但已不再关注的人群，其变化率取决于遗忘速度和自然衰减

**关键参数说明**

| 参数 | 含义 | 默认值 | 调整策略 |
|------|------|--------|----------|
| β（传播率） | 事件被传播的速度 | 0.3 | 负面情绪提升至1.2倍，正面情绪降至0.8倍 |
| γ（恢复率） | 人们遗忘事件的速度 | 0.1 | 根据事件类型动态调整 |
| μ（自然衰减率） | 信息的自然衰减 | 0.01 | 固定值 |

**计算流程**

1. **初始化**：设定初始人群分布（通常S=90%，I=10%，R=0%）
2. **情绪因子应用**：根据当前舆情情绪（正面/负面/中性/复杂）调整传播率β
3. **微分方程求解**：采用欧拉方法迭代求解每日的人群状态变化
4. **指标映射**：将感染人群比例I映射为热度值（0-100），结合易感人群比例S计算情绪指数
5. **风险分级**：根据热度值划分风险等级（高热>70为高风险，中热40-70为中风险，低热<40为低风险）

**输出结果**

模型输出包含未来N天的预测时间轴，每天包含：热度值、情绪指数、风险等级、事件阶段描述。该时间轴为后续的干预策略制定提供量化依据。

### 3.5.3.3 贝叶斯情景预测模型

贝叶斯情景预测模型用于生成舆情发展的多情景概率分布，帮助决策者了解不同发展路径的可能性。

**模型设计思路**

该模型基于贝叶斯概率理论，结合事件特征和当前情绪状态，动态计算各发展情景的概率分布。核心思想是：不同事件类型有其固有的发展规律，而当前情绪状态会影响这些规律的具体表现。

**预测流程**

1. **事件特征分析**：从事件摘要中提取关键特征，包括事件类型（产品问题/企业危机/公共安全/名人事件/政策/社会议题）、关键词、涉及主体等

2. **基础概率获取**：根据事件类型获取预设的基础情景概率分布。例如产品问题类事件的基础分布可能是：快速平息25%、持续发酵35%、局部爆发25%、全面危机15%

3. **情绪调整**：根据当前舆情情绪对基础概率进行动态调整。负面情绪会提升"全面危机"等高风险情景的概率，同时降低"快速平息"的概率；正面情绪则产生相反效果

4. **特征微调**：根据事件的具体特征（如涉及人数、传播渠道、历史类似事件等）进行概率微调

5. **概率归一化**：确保所有情景概率之和为100%，同时设置保底概率（5%）避免极端情况

**情景生成**

对于每个概率大于阈值的情景，模型会生成详细的情景描述，包括：
- 情景名称和发展路径描述
- 发生概率（百分比）
- 风险等级（高/中/低）
- 关键影响因素
- 预计时间线
- 核心洞察

**应用场景**

该模型输出的多情景概率分布，使决策者能够：
- 了解最可能发生的发展路径（概率最高的情景）
- 识别需要重点防范的低概率高风险情景
- 根据不同情景制定差异化的应对策略
- 持续跟踪实际发展与预测情景的偏差，及时调整策略

## 3.5.4 算法逻辑与约束机制（保留原有的代码层面约束）

### 3.5.4.1 ReAct 循环约束

ReAct循环通过多层约束确保推理过程的可靠性和完整性：

**工具调用次数约束**

每个分析模块在推理过程中需要满足工具调用的上下限要求：
- **上限约束**：单个模块最多调用5次工具，防止无限循环和资源浪费
- **下限约束**：单个模块至少调用2次工具，确保分析的充分性

当工具调用次数达到上限时，系统会强制输出当前结论；当未达到下限时，系统会提示继续调用工具以获取更充分的分析依据。

**必需工具检查**

不同分析模块有其必需的工具集合。例如风险评估模块必须使用风险评估工具，情景预测模块必须调用贝叶斯预测工具。如果最终答案中缺少必需工具的使用记录，系统会要求补充调用。

**循环终止条件**

ReAct循环在以下任一条件满足时终止：
1. 工具调用次数达到上限（强制终止）
2. 输出最终答案且满足最小工具调用次数
3. 输出最终答案且已使用所有必需工具

### 3.5.4.2 工具调用格式约束

为确保LLM输出的工具调用可被正确解析执行，系统定义了严格的格式规范：

**工具调用格式**

工具调用必须使用XML标签包裹，包含工具名称和JSON格式的参数：
```
<tool_call name="工具名称">
{"参数名": "参数值"}
</tool_call>
```

**最终答案格式**

最终答案需要以特定标识开头，并包含结论、数据和置信度：
```
Final Answer: {
    "结论": "...",
    "数据": {...},
    "置信度": 0.85
}
```

**格式验证流程**

每次LLM响应都会经过以下验证：
1. **正则匹配**：检查是否符合工具调用或最终答案的正则模式
2. **工具存在性**：验证调用的工具是否在可用工具列表中
3. **参数完整性**：检查JSON格式是否合法，必需参数是否齐全
4. **参数类型**：验证参数类型是否符合工具定义

验证失败时会向LLM返回错误提示，要求其重新生成符合格式的响应。

### 3.5.4.3 数据验证约束

预测结果输出前需要通过多层次的数据验证：

**时间轴约束**

预测时间轴的长度有明确限制：
- 最短1天，最长30天
- 超出范围会被视为无效预测

**情景数量约束**

多情景预测需要满足：
- 最少2个情景，最多5个情景
- 各情景概率之和必须等于100%
- 单个情景概率不能为0（保底5%）

**风险等级约束**

风险等级只能是预定义的三种之一：高风险（high）、中风险（medium）、低风险（low）。任何其他值都会被拒绝。

**置信度约束**

预测置信度必须在0到1之间，表示预测结果的可信程度。超出该范围的置信度会被截断到边界值。

**验证失败处理**

当验证发现错误时，系统会：
1. 收集所有错误信息
2. 将错误反馈给LLM要求修正
3. 最多重试3次，超过则使用默认值或降级方案

### 3.5.4.4 超时与重试约束

**工具调用超时**

不同工具根据其复杂度有不同的超时限制：

| 工具名称 | 超时时间 | 说明 |
|---------|---------|------|
| SIR模型分析 | 10秒 | 数学计算较快 |
| 贝叶斯预测 | 15秒 | 概率计算稍复杂 |
| 情绪分析 | 10秒 | 文本分析 |
| 风险评估 | 10秒 | 规则判断 |
| 干预模拟 | 20秒 | 需要多轮推演 |
| 深度检索 | 30秒 | 涉及外部API调用 |

**重试策略**

当工具调用失败时，系统采用指数退避策略进行重试：
- 最大重试次数：3次
- 初始重试间隔：2秒
- 退避倍数：2（即第2次等待4秒，第3次等待8秒）

**超时处理**

工具调用超时后：
1. 如果是可重试错误（如网络超时），触发重试机制
2. 如果重试耗尽，返回错误信息，LLM需调整策略或跳过该工具
3. 关键工具超时可能导致整个预测流程降级为简化模式

**核心实现机制**

### 3.5.5.2 轮询机制设计

系统针对不同场景设计了差异化的轮询策略，以平衡实时性和服务器负载。图谱构建进度查询采用2秒基础间隔配合指数退避策略，当数据连续多次无变化时逐步延长轮询间隔至最长10秒，一旦检测到数据更新立即重置为最短间隔。模拟准备进度采用阶段感知策略，在准备阶段保持2秒间隔，进入处理阶段后根据实际进度动态调整，完成后立即停止轮询。模拟运行状态对实时性要求最高，采用1秒间隔并优先使用WebSocket长连接，当WebSocket不可用时自动降级为轮询。报告生成进度采用3秒间隔配合章节级增量更新，仅传输新增内容减少数据量。Agent配置实时查看采用文件修改时间检测机制，通过对比时间戳避免不必要的请求。

智能轮询的核心机制包括数据变化检测、指数退避和精确睡眠。系统会缓存上次请求结果并与当前结果对比，数据变化时立即触发回调并重置轮询间隔，连续无变化时按退避系数逐步增加间隔直至上限。同时系统会精确计算请求耗时，从目标间隔中扣除实际耗时后进行睡眠，确保轮询间隔的准确性。当检测到任务完成或失败状态时，轮询自动终止释放资源。遇到请求错误时，系统会临时加倍轮询间隔避免对故障服务端造成压力，待恢复正常后逐步回到基础间隔。

### 3.5.5.3 Docker 容器化部署

系统采用 Docker Compose 进行容器化部署，前后端服务均使用 host 网络模式以简化通信配置。后端服务基于 Python 3.11 精简镜像构建，通过多步骤优化镜像体积：首先最小化安装系统依赖，然后利用 Docker 缓存层先复制依赖文件安装 Python 包，再复制应用代码。为保障数据持久化，将上传目录挂载到宿主机，环境变量以只读方式挂载。安全方面，创建非 root 用户运行应用，并配置 HTTP 健康检查实现自动故障恢复。资源限制上，后端服务限制使用 4 核 CPU 和 8GB 内存，预留 2 核 4GB 保证基础性能。

在工程化取舍上，选择 host 网络而非端口映射是为了避免复杂的网络配置；数据卷挂载确保容器重建后数据不丢失；未采用多阶段构建是因为 Python 依赖复杂，多阶段收益有限；健康检查机制则能及时发现服务异常并触发重启。

### 3.5.5.4 性能优化策略

系统从四个维度进行性能优化。异步处理方面，模拟运行等耗时操作采用线程池异步执行，主线程立即返回任务 ID，避免阻塞用户请求，客户端通过轮询获取进度。缓存策略采用两级缓存架构，L1 为内存缓存提供最快访问速度，L2 为本地文件缓存实现数据持久化，读取时优先访问 L1 未命中则查询 L2 并回填 L1。数据库层面配置 SQLAlchemy 连接池，设置 10 个基础连接和 20 个溢出连接，连接超时 30 秒，每小时回收空闲连接，并开启连接前 ping 检测避免使用失效连接。前端构建使用 Vite 进行代码分割，将第三方库和图表库分离为独立 chunk，启用 Terser 压缩并移除调试代码，同时预构建常用依赖加速开发服务器启动。
